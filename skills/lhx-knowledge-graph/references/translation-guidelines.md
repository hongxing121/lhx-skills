# 翻译指南

## 核心原则：素材有多长，决定翻译策略

| 素材类型 | 原文长度 | 翻译策略 | 中文目标长度 |
|---------|----------|---------|-------------|
| 股东信、Master Plan、备忘录 | <1万词 | **完整翻译** | 全文 |
| 长访谈（2-3 小时） | 2-9 万词 | **精华翻译** | 3000-6000 字 |
| 财报会议 | 1-3 万词（剥离后） | **精华翻译** | 1500-3000 字 |
| TED 演讲 | 5000-1万词 | **完整翻译** | 全文 |
| 著作章节 | 看情况 | 重要章节完整，次要章节提炼 | - |

## 完整翻译规则

- **意译，不要逐字翻译**——让中文读者读起来流畅自然
- **保留语气风格**：
  - 贝佐斯：温和理性、长期主义、偶有自嘲
  - 巴菲特：幽默自嘲、口语化、爱用类比
  - 马斯克：技术派、直接挑衅、偶尔反讽
  - 芒格：尖锐、极简、爱用反例
- **保留金句**——所有"会被引用的句子"都必须保留原意
- **数字和数据点要准确**——不能为了流畅而修改

## 精华翻译规则（长访谈和财报会议）

精华翻译不是简单的"翻译一部分"，是**主题重组**：

1. **跳过寒暄、客套、闲聊**
2. **跳过主持人的长篇问题**——只保留必要的提问语境（一句话即可）
3. **按主题归类而不是按时间顺序**——访谈常常话题跳跃，重新整理成"主题 1 / 主题 2 / 主题 3"
4. **每个主题保留**：
   - 主角的核心论述（2-4 段）
   - 1-2 个金句（用 blockquote 引用）
   - 具体案例或数据
5. **跳过技术细节冗余**——比如某个工程问题讲了 5 段，只留最有信息量的 1 段

### 精华翻译的目标长度参考

- 一场 5 万词的访谈 → 4000-5000 字精华
- 一场 9 万词的访谈 → 6000-8000 字精华（不能再多了）
- 一场 1 万词的财报会议 Musk 部分 → 2000 字精华

## wikilinks 标注规范

### 三类标注

1. **概念**：`[[第一性原理]]`、`[[长期主义]]`、`[[飞轮效应]]`
2. **公司/产品**：`[[Tesla]]`、`[[SpaceX]]`、`[[AWS]]`、`[[Prime]]`
3. **人物**：`[[贝佐斯]]`、`[[马斯克]]`、`[[芒格]]`

### 关键：用统一词表

启动并行翻译 Agent 时，**必须给所有 Agent 一份统一的 wikilink 词表**，否则会出现 `[[Tesla Roadster]]` 和 `[[Roadster]]` 这种命名漂移。

词表怎么来？
- 第一个项目（贝佐斯）：用户和 Claude 一起讨论确定
- 后续项目：可以参考第一个项目的概念列表，加上新人物的特色概念

词表大小约 30-50 个候选词足够，每个翻译 Agent 在标注 wikilinks 时优先用词表里的命名。

### 翻译完成后的去重检查

跑这个脚本看看有没有命名漂移：

```python
import re, os, glob
files = glob.glob(os.path.expanduser('~/Documents/Obsidian Vault/<人名>/<人名>/**/*.md'), recursive=True)
counts = {}
for f in files:
    with open(f, 'r') as fh:
        content = fh.read()
    links = re.findall(r'\[\[([^\]|]+)(?:\|[^\]]+)?\]\]', content)
    for l in links:
        l = l.strip()
        counts[l] = counts.get(l, 0) + 1
for k, v in sorted(counts.items(), key=lambda x: -x[1])[:80]:
    print(f'{v:4d}  {k}')
```

发现明显是同一个东西的不同写法，用 `find ... -exec sed -i ...` 批量替换归一。

## YAML frontmatter 规范

每篇翻译的 md 文件必须有 YAML frontmatter：

```markdown
---
title: "标题"
date: YYYY-MM-DD
author: 作者
type: [类型]  # 股东信 / Master Plan / 访谈 / 财报会议 / 概念 / 产品 / 人物 / 方法
tags: [类型, YYYY]
---
```

类型字段会被 build_site.py 用来生成顶部的彩色徽章。

## 文件命名规范

| 类型 | 命名格式 | 示例 |
|------|----------|------|
| 股东信 | `YYYY 作者致股东信.md` | `1997 贝佐斯致股东信.md` |
| Master Plan | `YYYY 总体规划 第X部.md` 或 `YYYY <英文标题>.md` | `2006 Master Plan Part 1.md` |
| 长访谈 | `YYYY-MM 节目名 集数 - 主题.md` | `2024-08 Lex Fridman 438 - Neuralink Team.md` |
| 财报会议 | `YYYY-Qx 公司 财报会议.md` | `2024-Q2 Tesla 财报会议.md` |
| 演讲 | `YYYY-MM 场所 - 主题.md` | `2017-04 TED - 我们正在建造的未来.md` |

**带日期前缀**很重要——侧边栏会按文件名字典序排序，日期前缀让时间顺序自然成立。
