# Agent 提示词模板

并行 Agent 是这个工作流的核心引擎。下面是 3 种 Agent 的标准提示词模板，每次启动时把变量替换掉就能用。

---

## 通用约定

每个 Agent 提示词都应该包含 4 个段落：

1. **任务声明**（一句话说清楚你要它干什么）
2. **输入/输出路径**（绝对路径）
3. **抓取/处理方式**（包括"抓取手段优先级"）
4. **格式规范**（YAML frontmatter、wikilinks、文件命名等）

---

## 模板 1：素材收集 Agent

每个收集 Agent 负责一类素材或一个时间段，并行启动 2-4 个。

```
你的任务是收集 [人物名] 的 [素材类型]。

需要获取以下 [N] 份素材：
1. [素材 1 标题]
   URL: [URL 1]
2. [素材 2 标题]
   URL: [URL 2]
...

**抓取手段优先级（重要，按顺序尝试）**：

1. **优先用 baoyu-url-to-markdown skill 加 `--headless`**：
   `npx -y bun ~/.claude/skills/baoyu-url-to-markdown/scripts/main.ts --headless <URL>`
   它用 headless Chrome（--headless=new）渲染，与真实 Chrome 渲染结果一致，能绕过 Cloudflare、tesla.com、seekingalpha.com 等大部分反爬。
   **`--headless` 必须加** —— 没加这个 flag 会反复弹出 Chrome 窗口打断用户的其他工作。
   输出会自动保存到 `url-to-markdown/<domain>/<slug>.md`，自带 YAML frontmatter（title/author/date）。

2. 如果上面失败，再用 WebFetch（适合简单静态页面）

3. 仍失败用 web.archive.org 套层：
   `https://web.archive.org/web/2020/<原 URL>`

4. 真的需要交互（登录、点弹窗等）才升级到 gstack/browse skill；这种情况无法 headless，必须用可见浏览器

5. 实在抓不到的可以跳过，最后告诉我哪些缺失

**输出格式**：
- 把所有抓到的内容汇总到一个文件：`/Users/hongxing/project/webchat/[人名拼音]-[类型].txt`
- 用 `=== [类型]: [标题] ([日期]) ===` 分隔每篇
- 如果总字数太大，分批保存到 `-1.txt`、`-2.txt`

**重要原则**：
- 必须是完整原文，不是摘要
- 如果是财报会议或类似有多人发言的素材，必须**剥离分析师问题和其他高管的发言**，只保留 [主角名] 自己的部分
- 每抓到一份就立即追加到目标文件，不要在内存里攒太多

完成后告诉我：
- 拿到了哪几份
- 没拿到的是哪几份、为什么
- 总文件大小
```

### 启动多个收集 Agent 的例子

**贝佐斯项目实例**（24 封股东信切成 3 批并行）：

- Agent 1: 1997-2004（8 封）
- Agent 2: 2005-2013（9 封）
- Agent 3: 2014-2023（10 封）

**马斯克项目实例**（3 类素材并行）：

- Agent 1: 3 个 Tesla Master Plan
- Agent 2: 12 场长访谈
- Agent 3: 10 场财报会议

## 模板 2：翻译 Agent

每个翻译 Agent 负责一个时间段或一类素材，并行启动 4-5 个。

```
你的任务是把 [素材类型] 翻译成中文，写入 Obsidian vault。

**素材文件**：`/Users/hongxing/project/webchat/[人名拼音]-[类型].txt`
搜索 "=== [类型]:" 找出每篇内容。

**输出目录**：`/Users/hongxing/Documents/Obsidian Vault/[中文名]/[中文名]/[源类型目录]/`

**你负责的范围**：[年份 1] 到 [年份 2]，共 [N] 篇

**翻译策略**：

[选 A 或 B]

**A：完整翻译（适合短文档：股东信、Master Plan、备忘录）**
- 全文翻译，保留 [人物] 的语气风格（比如：自信/幽默/技术派/温和）
- 意译而非逐字翻译，让中文读者读起来流畅自然

**B：精华翻译（适合长访谈、财报会议，原文超过 1 万词）**
- 跳过寒暄、客套、技术细节冗余
- 跳过主持人的长篇问题，只保留必要的提问语境
- 重点保留 [主角] 对核心议题的论述、金句、反常识观点
- 每篇控制在 [3000-6000] 字中文

**wikilinks 标注规范**：
识别下面这些**核心实体**，用 [[双括号]] 标注：

- **概念**：[[第一性原理]]、[[长期主义]]、[[飞轮效应]]、... （从下面词表里挑）
- **公司/产品**：[[Tesla]]、[[SpaceX]]、[[AWS]]、[[Prime]]、...
- **人物**：[[贝佐斯]]、[[马斯克]]、[[芒格]]、...

**词表（重要：用统一的命名，避免每个 Agent 各自创造不同名字）**：
- 概念：[列出 20-30 个候选概念]
- 公司：[列出 10-15 个候选公司]
- 人物：[列出 5-10 个候选人物]

**文件命名规范**：
- 股东信：`YYYY [作者]致股东信.md`
- 访谈：`YYYY-MM [节目名] [集数] - [主题].md`
- 财报会议：`YYYY-Qx [公司] 财报会议.md`
- Master Plan：`YYYY [Master Plan 标题].md`

**文件格式**：
```markdown
---
title: "标题"
date: YYYY-MM-DD
author: [作者]
type: [类型]
tags: [类型, YYYY]
---

# 标题

（中文翻译，标注 [[wikilinks]]）

---
**原文出处**：[URL]
```

请认真处理。先 Read 素材文件，然后逐篇用 Write 工具写入 md 文件。

完成后告诉我：
- 写入了哪些文件
- 每篇大概多少字
- 你提取了哪些新概念（如果你识别出词表里没有的重要概念，告诉我）
```

## 模板 3：知识抽取 Agent

并行启动 3-4 个：概念抽取 / 产品抽取 / 人物抽取 /（可选）方法抽取。

```
你的任务是为 [人物] 知识库构建 [N] 张 [概念/产品/人物/方法] 卡片。

**素材路径**：`/Users/hongxing/Documents/Obsidian Vault/[中文名]/[中文名]/`
- `[源类型 1]/` - X 篇素材
- `[源类型 2]/` - Y 篇素材
- ...

**输出路径**：`/Users/hongxing/Documents/Obsidian Vault/[中文名]/[中文名]/[concepts|companies|people|methods]/`

**需要建立的 [N] 张卡片**：

[列出每张卡片的标题 + 一句话描述]

例如概念卡片：
1. **第一性原理.md** (First Principles Thinking) — 从物理学根基拆解问题
2. **白痴指数.md** (Idiot Index) — 零部件实际成本 vs 理论原材料成本
...

例如方法卡片：
1. **冲刺模式.md** (Surge Mode) — 集中一段时间专注解决一个大问题
2. ...

**每张卡片的结构**：

```markdown
---
title: "卡片名（English Name）"
type: [概念|产品|人物|方法]
tags: [type]
---

# 卡片名（English Name）

## [适用类型的章节标题，如"概念解析"或"基本信息"]

### 定义与起源
（这个东西最早在哪场访谈/股东信/Master Plan 出现，[主角] 怎么定义的）

### 核心要义
（3-4 个子论点，每个配原文引用和具体出处链接，例如 [[2021-12 Lex Fridman 252]]）

### 实践应用
（具体到哪个产品、什么年份、什么数据）

### 常见误区
（对这个东西常见的误解）

## [主角] 原话精选
（3-5 条原话引用，每条标注出处 [[文件名]]）

## 思想演变（如适用，但对"立场演变"类卡片是必填）
（按时间阶段追踪它在不同素材里的演变）

## 相关 [概念/产品/人物]
（[[双括号]] 链接其他卡片）

## 典型案例
（涉及的产品/公司/项目案例）
```

**特殊要求**：

[可选 - 仅用于方法卡片]
方法卡片必须额外有一节 "**你能用上吗？**"——讲普通团队/创业者怎么借鉴这个方法。这是方法卡相对于概念卡的差异化价值。

[可选 - 仅用于"立场演变"类概念]
立场演变卡片要按时间线整理变化轨迹，最好用表格呈现每个阶段的立场和触发事件。

**重要原则**：

1. **必须从素材里提取真实内容**，不要凭空编造引用
2. **引用必须有出处链接**，用 `[[文件名]]` 格式（不是 URL）
3. **如果某张卡片在素材里没有直接对应的内容**：可以用公开常识补充背景，但必须在卡片末尾诚实标注"本卡片部分内容基于公开资料补充，素材中无直接引用"
4. **用 [[双括号]] 链接到其他卡片**，建立交叉引用
5. **先 Read 素材文件**理解内容，再批量写入

完成后告诉我：
- 写入了哪几张卡片
- 每张卡片的核心引用来源（哪几篇素材）
- 哪些卡片在素材里证据较弱、需要补充
```

## 启动并行 Agent 的核心原则

1. **永远不要串行做大批量工作**。如果有 20 个文件要写，分给 4 个 Agent 每人 5 个，比一个 Agent 写 20 个要快得多
2. **每个 Agent 都要给绝对路径**，不要让它们猜
3. **不同 Agent 的输出目录可以有重叠**，但要在 prompt 里明确告诉它"如果发现某个文件已经存在，跳过"
4. **完成后必须 ls 检查并去重**——并行 Agent 偶尔会用不同的命名约定写同一个文件（如 "2019-04 Lex Fridman.md" vs "Lex Fridman 49 (2019).md"）
5. **让 Agent 报告它做了什么**——这不只是为了你看，更是为了后续 Agent 能引用前一个 Agent 的输出

---

## 失败处理

如果某个 Agent 触发了 rate limit 或上下文超限：

1. **不要重新启动同一个 Agent 做同样的事**——可能是任务太大
2. **拆得更细**：把任务分成更小的批次，启动新 Agent 接力做剩下的
3. **检查它已经写入的文件**：很多时候 Agent 在 fail 之前其实已经完成了一大半，不要重复劳动
